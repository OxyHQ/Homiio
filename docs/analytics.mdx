# Analytics

Homiio ships two analytics surfaces, both server-side:

1. **Per-user insights** — `GET /api/analytics` returns the calling user's interaction summary, cross-app actions, and trend deltas. When the optional `horizonService` is available it delegates to the Oxy ecosystem cross-app analytics layer; otherwise it returns a deterministic fallback so the Insights screen never has to render an error state.
2. **Public marketplace stats** — `GET /api/analytics/stats` runs four MongoDB aggregations in parallel and powers the homepage's "market at a glance" widgets.

The corresponding controller lives at `packages/backend/controllers/analyticsController.ts`. The Insights screen at `packages/frontend/app/insights/index.tsx` consumes both endpoints through `packages/frontend/services/analyticsService.ts` over TanStack Query, with charts rendered via `react-native-chart-kit` over `react-native-svg`.

There is currently no client-side product-analytics SDK (PostHog, Amplitude, Mixpanel) wired into the frontend tree. All analytics in production are server-aggregated from MongoDB.

## `GET /api/analytics`

Authenticated. Returns the user's interaction summary, cross-app actions, and trend deltas:

```ts
interface AnalyticsInsights {
  totalInteractions: number;
  appsUsed: string[];
  crossAppActions: number;
  dataShared: number;
  insights: string[];
  trends: {
    propertyRevenue: { current: number; change: number; period: string };
    tenantSatisfaction: { current: number; change: number; period: string };
  };
}
```

The controller first tries to delegate to the `horizonService` (`packages/backend/services`). When that service is missing — common in standalone dev — it returns a fallback payload tagged `'Analytics retrieved successfully (fallback data)'`. The `period` query param accepts `7d`, `30d`, or `90d` and defaults to `30d`.

## `GET /api/analytics/stats`

Public — mounted under `routes/public.ts`. Runs four parallel Mongo aggregations:

```ts
const [totalProperties, totalCities, totalSaves, uniqueSavers] = await Promise.all([
  Property.countDocuments({}).catch(() => 0),
  City.countDocuments({}).catch(() => 0),
  Saved ? Saved.countDocuments({ targetType: 'property' }).catch(() => 0) : 0,
  Saved
    ? Saved.distinct('profileId', { targetType: 'property' }).then(arr => arr.length).catch(() => 0)
    : 0,
]);
```

Followed by a pricing aggregate (`averageRent`, `minRent`, `maxRent`), a top-cities aggregate joining `addresses` via `$lookup`, and a price-bucket histogram with preset boundaries:

```ts
$bucket: {
  groupBy: '$rent.amount',
  boundaries: [0, 500, 1000, 1500, 2000, 3000, 5000, 10000],
  default: '10000+',
  output: { count: { $sum: 1 } },
}
```

The response feeds the public homepage widgets:

```ts
interface AppStats {
  totals: { properties: number; cities: number; saves: number; uniqueSavers: number };
  pricing: { averageRent: number; minRent: number; maxRent: number };
  topCities: { city: string; state: string; properties: number; averageRent: number }[];
  priceBuckets: { bucket: string; count: number }[];
}
```

Every aggregate is wrapped in `.catch(() => 0)` (or `[]`) and the entire handler has a top-level fallback so a single failing pipeline never produces a 500 — partial data is always preferable to an outage on a public marketing page.

## Frontend service wrapper

`packages/frontend/services/analyticsService.ts` calls both endpoints and provides identical fallback shapes:

```ts
async getAnalytics(period: '7d' | '30d' | '90d' = '30d'): Promise<AnalyticsInsights> {
  try {
    const response = await api.get<{ success: boolean; data: AnalyticsInsights }>(
      this.baseUrl,
      { params: { period } },
    );
    return (response.data as AnalyticsApiResponse).data ?? response.data;
  } catch {
    return FALLBACK_INSIGHTS;
  }
}
```

The Insights screen consumes both methods via TanStack Query so they participate in the global query cache, optimistic refetch on focus, and offline-first replay.

## Privacy posture

- `GET /api/analytics` returns only the calling user's data. The controller scopes off `req.userId` (set by the `oxy.auth()` middleware) and accepts a `userID` query override only for delegated admin calls.
- `GET /api/analytics/stats` is intentionally public and returns only marketplace-level aggregates — no per-user information leaves the server.
- No third-party analytics SDK is loaded on web or native, so there is no client-side IP, fingerprint, or session replay leaving the device.
